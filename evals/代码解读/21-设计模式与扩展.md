# 设计模式与扩展

从构建"类似系统"的角度，梳理 `evals/` 目录中的关键设计模式和可复用经验。

---

## 一、评估套件的标准结构

每个套件都遵循同一个模板，降低新增套件的认知成本：

```
suite-name/
├── package.json        ← 依赖 @deepagents/evals
├── vitest.config.ts    ← 固定模板，配置 LangSmith reporter
├── eval.test.ts        ← 两行代码：获取 runner + 调用套件函数
└── index.ts            ← 导出一个 {name}Suite(runner) 函数
```

### 标准 eval.test.ts（固定模式）

```typescript
import { getDefaultRunner } from "@deepagents/evals";
import { mySuite } from "./index.js";

const runner = getDefaultRunner();
mySuite(runner);
```

### 标准 index.ts（套件函数模式）

```typescript
import { ls } from "langsmith/vitest";
import type { EvalRunner } from "@deepagents/evals";

export function mySuite(runner: EvalRunner): void {
  ls.describe("my-suite-name", () => {
    ls.test("test case name",
      { inputs: { query: "..." } },
      async ({ inputs }) => {
        const result = await runner.run({ query: inputs.query });
        expect(result).toHaveFinalTextContaining("expected");
        ls.logFeedback({ key: "agent_steps", score: result.steps.length });
      }
    );
  });
}
```

**为什么用函数而不是直接在文件顶层调用？**
→ `all` 套件需要导入并组合所有套件函数，函数形式使组合变得简单。

## 二、Runner 扩展模式

`runner.extend(overrides)` 是最常用的配置注入方式，支持**临时覆盖**而不影响其他测试：

```typescript
// 每次 extend 返回新 Runner，原 Runner 不变
const baseRunner = getDefaultRunner();

// 覆盖系统提示
const customRunner = baseRunner.extend({ systemPrompt: "You are Alice." });

// 覆盖工具集
const toolRunner = baseRunner.extend({ tools: [myTool] });

// 覆盖多个配置
const fullRunner = baseRunner.extend({
  systemPrompt: "...",
  tools: [myTool],
  memory: ["/AGENTS.md"],
  skills: ["/skills/"],
  subagents: [{ name: "helper", ... }],
});

// 链式扩展（后者覆盖前者）
const chainedRunner = baseRunner
  .extend({ systemPrompt: "Base prompt" })
  .extend({ tools: [extraTool] });
```

## 三、Spy 工具模式（可验证的工具）

当需要断言工具是否被调用、以什么参数被调用时，用 `vi.fn` 包装工具函数：

```typescript
import { vi } from "vitest";

function spiedTool<TInput>(
  fn: (input: TInput) => Promise<string>,
  config: ToolConfig
): { tool: StructuredTool; spy: ReturnType<typeof vi.fn> } {
  const spy = vi.fn(fn);
  return { tool: tool(spy, config), spy };
}

// 使用
const { tool: myTool, spy: myToolSpy } = spiedTool(
  async ({ arg }) => `Result: ${arg}`,
  { name: "my_tool", schema: z.object({ arg: z.string() }) }
);

// 断言
expect(myToolSpy).toHaveBeenCalled();
expect(myToolSpy).toHaveBeenCalledWith(
  expect.objectContaining({ arg: "expected" }),
  expect.anything()  // LangChain config 参数，用 anything() 忽略
);

// 每个测试后重置（避免测试间污染）
afterEach(() => myToolSpy.mockClear());
```

## 四、数据驱动测试模式

对于大量相似测试用例，用数组 + 循环代替重复代码：

```typescript
const CASES = [
  { name: "case 1", query: "...", expectedText: "...", expectedTool: "..." },
  { name: "case 2", query: "...", expectedText: "...", expectedTool: "..." },
  // ...
];

ls.describe("my-suite", () => {
  for (const testCase of CASES) {
    ls.test(testCase.name,
      { inputs: { query: testCase.query } },
      async ({ inputs }) => {
        const result = await runner.run({ query: inputs.query });
        expect(result).toHaveFinalTextContaining(testCase.expectedText);
      }
    );
  }
});
```

**进阶：带 mustContainOneOf/mustNotContain 的断言模式**（来自 followup-quality）：

```typescript
interface TestCase {
  mustContainOneOf: string[];
  mustNotContain?: string[];
}

const finalText = getFinalText(result).toLowerCase();

const hasAny = testCase.mustContainOneOf.some(kw => finalText.includes(kw));
expect(hasAny).toBe(true);

for (const forbidden of testCase.mustNotContain ?? []) {
  expect(finalText).not.toContain(forbidden);
}
```

## 五、文件预置模式

通过 `initialFiles` 为 Agent 预置虚拟文件系统内容：

```typescript
runner.run({
  query: "...",
  initialFiles: {
    "/data/config.json": JSON.stringify({ key: "value" }),
    "/docs/readme.md": "# Project\n...",
    "/AGENTS.md": "User prefers TypeScript.",
  }
});
```

**注意**：`initialFiles` 的值是**字符串**，harness 内部会转换为 LangGraph 的 `FileData` 格式（带时间戳）。

## 六、效率断言模式

评估 Agent 是否采用了高效策略（而非正确但低效的做法）：

```typescript
// 断言并行操作（特定步骤有多个工具调用）
const step1 = result.steps[0];
expect(step1.action.tool_calls.length).toBeGreaterThan(1);

// 断言步骤数（间接验证效率）
expect(result).toHaveAgentSteps(2);  // 不应该超过 2 步

// 断言特定步骤使用了高效工具（如 grep 而非逐个 read）
expect(result).toHaveToolCallInStep(1, { name: "grep" });
```

## 七、LangSmith Feedback 规范

统一的 feedback 命名便于跨套件对比：

```typescript
// 通用 feedback（建议每个套件都上报）
ls.logFeedback({ key: "agent_steps", score: result.steps.length });

// 套件特定 feedback
ls.logFeedback({ key: "case_name", value: "specific_case" });
ls.logFeedback({ key: "difficulty", value: "hard" });

// 自定义指标
ls.logFeedback({ key: "accuracy", score: 0.85 });  // 0~1
ls.logFeedback({ key: "tool_calls", score: totalToolCalls });
```

## 八、如何新增一个评估套件

### 步骤

1. **创建套件目录**

```bash
mkdir -p evals/my-new-suite
cd evals/my-new-suite
```

2. **创建 package.json**

```json
{
  "name": "@deepagents/evals-my-new-suite",
  "version": "0.0.0",
  "private": true,
  "scripts": {
    "test:eval": "vitest run"
  },
  "dependencies": {
    "@deepagents/evals": "workspace:*"
  }
}
```

3. **创建 vitest.config.ts**（复制任意套件的配置文件）

4. **创建 index.ts**

```typescript
import { ls } from "langsmith/vitest";
import type { EvalRunner } from "@deepagents/evals";

export function myNewSuite(runner: EvalRunner): void {
  ls.describe("my-new-suite", () => {
    ls.test("first test",
      { inputs: { query: "Tell me a joke" } },
      async ({ inputs }) => {
        const result = await runner.run({ query: inputs.query });
        expect(result).toHaveFinalTextContaining("?");  // 笑话通常有问号
        ls.logFeedback({ key: "agent_steps", score: result.steps.length });
      }
    );
  });
}
```

5. **创建 eval.test.ts**

```typescript
import { getDefaultRunner } from "@deepagents/evals";
import { myNewSuite } from "./index.js";

const runner = getDefaultRunner();
myNewSuite(runner);
```

6. **（可选）在 all/eval.test.ts 中添加引用**

```typescript
import { myNewSuite } from "@deepagents/evals-my-new-suite";
myNewSuite(runner);
```

## 九、如何添加新的 Runner（新模型）

在 `internal/eval-harness/src/setup.ts` 中添加：

```typescript
registerDeepAgentRunner("gemini-2.0-flash", (config) =>
  createDeepAgent({
    model: "google:gemini-2.0-flash",
    ...config,
  })
);
```

然后通过 `EVAL_RUNNER=gemini-2.0-flash` 运行。

## 十、关键经验总结

| 经验 | 说明 |
|------|------|
| **函数形式套件** | `export function xxxSuite(runner)` 使组合变得简单 |
| **Runner 扩展而非修改** | `extend()` 保持原 Runner 不变，测试隔离 |
| **Spy 工具** | 用 `vi.fn` 包装使工具调用可验证 |
| **数据驱动** | 大量相似用例用数组+循环，不重复代码 |
| **效率也是测试目标** | 不仅测正确性，也测并行能力和步数 |
| **mustNotContain** | 比 mustContain 更严格，验证 Agent 的"不该做什么" |
| **统一 feedback** | `agent_steps` 在所有套件上报，便于横向对比 |
| **oolong 单独运行** | 高成本套件默认不加入 all，避免拖慢日常 CI |
